# v0.7.2 Changelog Draft

Status: internal draft. Do not publish until the v0.7.2 release cut.

This draft records the three implemented v0.7.2 slices so the final public
`CHANGELOG.md` can be written from implementation facts, not memory.

## [0.7.2] — Unreleased

### Added

- **Reader-final output gate**: `finalize` now runs a deterministic reader-facing cleanliness gate and writes `output/intermediate/finalize_report.json` with `reader_clean` status and residue counts.
- **Reader residue detection**: final reader artifacts are checked for internal source markers, raw claim IDs, source IDs, local paths, debug residue, workflow/process wording, and blank citation/source-index rows.
- **DOCX reader scanning**: reader-final checks inspect DOCX body text plus headers and footers, so process residue in rendered delivery files is not hidden from the final gate.
- **Runtime completion transactions**: added `multi-agent-brief state stage-complete` and `multi-agent-brief state finalize-complete` as deterministic transaction commands for successful stage completion and terminal finalize recording.
- **Finalize transaction enforcement**: `finalize-complete` requires valid quality gate report, no blocking gate findings, passing `finalize_report.json`, passing reader-clean checks, and a valid reader brief before terminal state can be recorded.
- **Runtime manifest extension preservation**: runtime state refresh and completion transactions preserve registered extension blocks such as `runtime_manifest.improvement` and `runtime_manifest.recipe`, preventing read/refresh paths from silently dropping materialization evidence.
- **Claude Code five-verb writer entrypoint**: added `/mabw` with the writer-facing verbs `new`, `run`, `status`, `feedback`, and `deliver` for Claude Code.
- **Claude install support for `/mabw`**: `multi-agent-brief claude install` installs `/mabw` alongside `/generate-brief` and MABW subagents.
- **Control-surface documentation**: added public control-surface notes documenting what MABW tracks, who writes each surface, and how runtime records, evidence artifacts, gates, memory, and repo contracts are separated.
- **Improvement Ledger supersession support**: added top-level immutable `supersedes_id` lineage for approved reader-guidance replacements.
- **Improvement duplicate warnings**: `improve propose` now emits deterministic same-category/scope duplicate warnings, and `--supersedes` proposals against already-superseded targets return `approval_blocker=true`.
- **Improvement fork protection**: approved supersession forks are rejected with `supersession_fork`, so memory/materialization never chooses between two approved replacements for the same target.
- **Improvement materialization hygiene**: non-materializable superseders surface deterministic warnings, and reverting a superseder warns when an old approved entry becomes materializable again.
- **Improvement origin runtime rendering**: `origin_runtime` remains audit/rendering metadata only and can appear in memory source lines without affecting routing, filtering, or materialization.

### Changed

- **Stage success path moved away from raw `state decide`**: normal successful stage completion now uses `state stage-complete`; raw `state decide` remains for lower-level retry, repair, human-review, and block decisions.
- **Final delivery path hardened**: delivery guidance now runs gates, strict state check, `stage-complete`, `finalize`, and `finalize-complete` in order rather than treating `finalize` as a quality-gate executor.
- **Runtime handoff and agent instructions hardened**: Orchestrator-facing guidance across Claude, OpenCode, Codex, Hermes, and shared role sources now points to completion transactions for success paths.
- **Reader-output failure is explicit**: finalizer reports reader-clean failures through `finalize_report.json` and exits cleanly with actionable failure output instead of silently shipping process residue.
- **Public quickstart shifted to Claude Code first**: README, README_en, support matrix, and Claude quickstart now present Claude Code as the first runtime for the five-verb product path.
- **Five-verb product boundary documented**: the five-verb `/mabw` entrypoint first ships only on Claude Code; Hermes, OpenCode, Codex, and manual runtimes keep their existing entrypoints to avoid a false parity contract.

### Fixed

- **Process wording leakage**: reader-facing briefs are checked for wording such as internal workflow, Claim Ledger, artifact registry, runtime state, and quality gate terms that should not appear in final user output.
- **Bare claim/source ID leakage**: reader-facing Markdown and DOCX outputs now detect raw internal identifiers such as claim IDs and source IDs even when they are not in `[src:...]` marker form.
- **Blank citation table rows**: source/citation sections with blank ID/source/reference cells are detected as reader-clean failures.
- **Duplicate process-wording counts**: reader-clean counts avoid double-counting the same process wording occurrence.
- **Manifest extension loss**: completion/state paths defend against the v0.7.1 failure mode where a runtime-state refresh rewrote the manifest and dropped the `improvement` block.

### Boundaries

- v0.7.2 does not reintroduce a Python brief-generation pipeline. Python validates, records, gates, renders, and transacts control state; the external runtime still performs Orchestrator and specialist-role work.
- v0.7.2 does not make `/mabw` a cross-runtime parity promise. The five-verb writer entrypoint is Claude Code first; other runtimes keep their existing workflow surfaces.
- v0.7.2 does not claim output quality improvement. Reader-final gates only reject known process/residue defects; they do not judge prose quality or semantic usefulness.
- v0.7.2 does not add autonomous repair, automatic learning, automatic approval, retrieval memory, mode registry, or runtime-specific guidance filtering.
- v0.7.2 includes the Improvement Ledger schema hygiene batch (`supersedes_id`, duplicate warnings, approved fork rejection, computed supersession, and `origin_runtime` rendering hygiene).
- v0.7.2 does not include `operator_reported_model`; that runtime observation metadata is deferred to v0.7.3 / v0.8 scorecard design.
- v0.7.2 does not include intake/candidates parking-lot skeletons.

## Slice Summary

### PR1 — Reader-Final Output Gate

Purpose: stop final delivery from leaking internal process residue into reader-facing outputs.

Implemented:

- `src/multi_agent_brief/outputs/reader_final_gate.py`
- `finalize_report.json.reader_clean`
- raw claim/source ID detection;
- `[src:...]` and malformed source marker detection;
- local path and debug residue detection;
- English/Chinese process wording detection;
- blank citation/source-index row detection;
- DOCX body/header/footer scanning;
- finalize tests covering these failure modes.

Architecture meaning:

- The final reader output becomes its own gate surface, separate from audit and claim support.
- `finalize` may render, but cannot silently call residue-clean output "clean".
- This is deterministic delivery hygiene, not semantic judging.

### PR2 — Runtime Completion Transactions

Purpose: move fragile Orchestrator bookkeeping from prompt obedience into deterministic transactions.

Implemented:

- `state stage-complete`;
- `state finalize-complete`;
- transaction error codes and JSON output;
- atomic validation-before-write behavior;
- duplicate/replay rejection;
- final gate and quality gate enforcement before terminal finalize state;
- preservation of registered manifest extension keys;
- updated handoff/runtime guidance across generated role assets and Hermes surfaces;
- eval-case support for completion actions and manifest assertions.

Architecture meaning:

- "A stage is complete" becomes one transaction, not a chain of things an LLM must remember to update.
- The Orchestrator keeps semantic agency; Python owns deterministic bookkeeping.
- This implements the rule: active intelligence remains outside Python, but the ledger belongs to Python.

### PR3 — Claude Five-Verb Product Entrypoint

Purpose: expose the product through a small writer-facing vocabulary without turning every runtime into a mirrored UX surface.

Implemented:

- `.claude/commands/mabw.md`;
- first-screen verbs: `new`, `run`, `status`, `feedback`, `deliver`;
- `/mabw new` creates workspace/handoff but does not generate a brief;
- `/mabw run` refreshes Claude handoff but does not execute stages;
- `/mabw status` is specified as read-only dashboard behavior;
- `/mabw feedback` records human feedback and requires confirmation for downstream actions;
- `/mabw deliver` routes delivery through gates, state check, completion transactions, finalize, and finalization transaction;
- `multi-agent-brief claude install` installs `/mabw`;
- README/README_en/support matrix/Claude quickstart update the first-class path.

Architecture meaning:

- The writer-facing product surface is five verbs, not the whole CLI inventory.
- `doctor` remains diagnostic/maintainer tooling, not a sixth writer verb.
- Claude Code is the first implementation target because the reference runs were actually executed there; parity is deliberately deferred.

### PR4 — Improvement Schema Hygiene

Purpose: make approved reader-guidance replacement explicit without adding a
fifth ledger state or semantic duplicate detection.

Implemented:

- top-level immutable `supersedes_id`;
- missing historical `supersedes_id` treated as semantically null;
- deterministic same category/scope proposal warning;
- proposed supersession forks allowed with `approval_blocker=true` warning;
- approved supersession forks rejected with `supersession_fork`;
- materialization excludes computed-superseded approved entries;
- revert of a superseder can re-expose the old approved entry and returns a warning;
- non-materializable superseder suppression warning;
- no stored `superseded` status;
- `origin_runtime` remains rendering/audit metadata only.

Architecture meaning:

- Supersession is entry-level lineage, not mutable change text.
- `supersede != retire`: if the old entry is wrong rather than replaced, the operator explicitly reverts it.
- Python computes deterministic lineage and materialization eligibility; humans still write and approve guidance.
- `operator_reported_model` is deferred to v0.7.3 / v0.8 scorecard metadata.
