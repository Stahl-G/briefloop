# v0.7.2 PR4 — Improvement Schema Hygiene Batch

## Summary

PR4 is the v0.7.2 schema hygiene batch for the Improvement Ledger.

It pays back the v0.7.1 dogfooding findings:

- duplicate active guidance can accumulate because `supersedes_id` was cut from
  the v0.7.0 implementation;
- operators need a deterministic warning when they are about to create another
  active guidance entry with the same category/scope;
- existing `origin_runtime` provenance should remain audit/rendering metadata
  only, not a routing or filtering mechanism.

This PR does **not** add a fifth ledger state, semantic similarity matching,
automatic learning, runtime-specific filtering, intake/candidates parking lots,
or new output-quality claims.

## Architecture Rule

Apply the standing authority model:

> 聪明的无权，有权的确定，生效的过人，过人的留痕。

For PR4:

- LLM/user text may propose a replacement, but cannot silently mutate existing
  guidance.
- Python validates references, duplicate warnings, cycles, materialization
  eligibility, and approved-supersession fork prevention.
- Human approval remains the only promotion gate.
- Supersession is append-only evidence, not an in-place edit.
- Derived labels such as "superseded" are computed read views, never stored
  lifecycle states.

Also keep the two hardening rules:

- 同一个字段只许有一个写者。
- 写成机器的部分没有坏，托付给指令的部分全坏了。

## Scope

Implement four bounded changes:

1. Add optional `supersedes_id` to Improvement Ledger entries.
2. Add deterministic duplicate warning on `improve propose`.
3. Preserve and test `origin_runtime` as provenance/rendering metadata only.
4. Record the decision to defer `operator_reported_model` to v0.7.3 / v0.8
   scorecard metadata.

Do not split items 1 and 2; supersession without duplicate warning leaves the
operator blind.

## Non-Goals

- No `superseded` stored status.
- No `partial`, `mixed`, `weakened`, `sharpened`, or `effective` ledger status.
- No semantic duplicate detection or embedding similarity.
- No automatic revert of superseded entries.
- No automatic approval.
- No runtime-specific materialization filtering.
- No generic unbounded `provenance` blob.
- No `improvement/intake.jsonl` or `improvement/candidates.jsonl`; those are a
  separate intake slice if still needed.
- No public claim that memory reliably improves output quality.

## Ledger Schema Decision

### Field

Add top-level optional field:

```json
"supersedes_id": "AG-0001"
```

Rules:

- New writer output always includes `supersedes_id`, either a valid entry id or
  `null`.
- Existing historical entries may omit the field and remain valid.
- `revision = 1` may introduce `supersedes_id`.
- Later revisions must copy it unchanged.
- `supersedes_id` is part of the canonical revision hash when present.
- `supersedes_id` must match `^AG-\d{4,}$` when non-null.
- `supersedes_id` must not equal the entry's own `entry_id`.
- Ledger-level validation rejects unknown references and supersession cycles.
- Ledger-level validation rejects approved supersession forks: at most one
  current approved entry may supersede a given target.

Rationale:

- Top-level placement makes supersession a ledger lifecycle relation, not part
  of the free-text `change`.
- Keeping it immutable across revisions preserves the "complete immutable entry
  revision" model.
- Accepting absence only for historical entries avoids a breaking migration.

### Computed Supersession

Do not add a lifecycle state.

Compute read-view supersession from current ledger state:

```text
entry A is superseded if there exists current entry B such that:
  B.status == approved
  B.supersedes_id == A.entry_id
```

Consequences:

- A superseded entry remains whatever status its current revision says.
- `improve list/show/stats/validate` may display `superseded_by` as a computed
  field.
- `improvement/memory.md` and runtime snapshots include only approved,
  materializable, non-superseded entries.
- If the superseding entry is later reverted, the old entry becomes eligible
  again if it is still approved and materializable.
- No automatic revert is written to the old entry.
- `supersede != retire`: if the old entry is wrong rather than merely replaced,
  the operator must explicitly revert the old entry too.
- Chains resolve without recursion in the materializer:
  `C supersedes B supersedes A` outputs C; reverting C re-exposes B.
- An approved but non-materializable superseder still suppresses its target. That
  is a deterministic materialization warning, not a reason to change the
  predicate.

This keeps "application" and "current materialization eligibility" as per-run /
computed facts, not ledger state.

## CLI/API Decision

### `improve propose`

Add:

```bash
multi-agent-brief improve propose \
  --workspace <ws> \
  --guidance <text> \
  --category <enum> \
  --scope <enum> \
  --source-summary <text> \
  [--supersedes AG-0001] \
  [--json]
```

For `--from-issue`, the same `--supersedes` option is allowed, but it remains
evidence-only: the issue does not auto-promote into guidance.

`--supersedes` rules:

- target entry must exist in the current ledger;
- target current **stored** status must be `approved`;
- target must be materializable under product-definition rules;
- category/scope mismatch is allowed only with a warning, not a block, because
  the operator may intentionally replace a broader/narrower rule;
- target already computed-superseded by another approved current entry is still
  accepted at proposal time, but must return a deterministic warning with
  `approval_blocker: true`;
- approval fails with `supersession_fork` until the existing approved superseder
  is reverted;
- memory/materialization never chooses between two approved replacements because
  the ledger contract disallows approved forks;
- self-supersession and cycles are errors.

The command appends only the new proposed revision. It does not modify the old
entry.

On revert of a superseding entry, emit a deterministic non-blocking warning for
each entry that becomes materializable again:

```text
reverting AG-0002 re-exposes AG-0001
```

This warning is computed at command time and is not stored as ledger state.

### Duplicate Warning

On propose without `--supersedes`, compute deterministic duplicate warnings:

```text
same target_kind
same level
same change.category
same change.scope
current status in proposed/approved
not computed-superseded
```

Return a warning with code:

```text
possible_duplicate_active_guidance
```

The warning is not blocking. Human approval remains the gate.

No semantic matching, no embeddings, no "similar wording" heuristic.

JSON output should include:

```json
"warnings": [
  {
    "code": "possible_duplicate_active_guidance",
    "entry_id": "AG-0001",
    "message": "Existing active guidance has the same category/scope. Consider --supersedes AG-0001."
  }
]
```

Human output should say the same in one line.

### Validate/List/Show/Stats

Add computed read fields:

- `supersedes_id`
- `superseded_by`
- `is_superseded`
- `materializable`
- `non_materializable_reason`

`stats` should count:

- `approved_count`
- `superseded_count`
- `eligible_for_materialization_count`

`eligible_for_materialization_count` must exclude computed-superseded entries.

Do not rename it to "active applied" or "effective".

## Runtime Observation Schema

Decision: defer `operator_reported_model` to v0.7.3 / v0.8 scorecard metadata.

Reason:

- PR4 already closes Improvement Ledger schema hygiene.
- `operator_reported_model` is runtime observation metadata, not Improvement
  Ledger hygiene.
- Adding it touches run/start/runtime manifest surfaces and risks delaying
  v0.7.2.
- It should be designed with reference-run / v0.8 scorecard metadata.

For v0.7.2, `origin_runtime` remains the only PR4 provenance field currently
implemented. `origin_runtime` is audit/rendering metadata only, not
routing/filtering/materialization.

## `origin_runtime` Regression Contract

`source_evidence[].origin.origin_runtime` is already implemented.

PR4 should only lock the semantics:

- render in `improvement/memory.md` when present;
- include in ledger validation/hygiene;
- never use for materialization filtering;
- never use for runtime routing;
- never use for policy decisions.

Add regression tests if any path currently lacks them.

## Materialization Update

Update the memory projection selection:

```text
selected = approved
  AND product-definition materializable
  AND not status-suppressed-by-approved-superseder
```

Stable ordering remains by `entry_id`.

Skipped diagnostics should include superseded entries:

```json
{
  "entry_id": "AG-0001",
  "reason_code": "superseded_by_active_guidance",
  "message": "Entry is superseded by AG-0002."
}
```

This diagnostic appears in rebuild/freeze outputs, not in the ledger.

If an approved non-materializable entry suppresses a target, emit a deterministic
warning:

```json
{
  "entry_id": "AG-0002",
  "reason_code": "non_materializable_superseder_suppresses_target",
  "message": "AG-0002 suppresses AG-0001 but is not materializable."
}
```

The warning makes the coverage gap visible without inventing another state.

## Tests

### Contract Tests

- old entries without `supersedes_id` remain valid.
- new writer-style entries with `supersedes_id: null` are valid.
- valid `supersedes_id` is accepted.
- bad id format is rejected.
- self-supersession is rejected.
- unknown referenced entry is rejected in ledger-level validation.
- supersession cycle is rejected.
- later revision changing `supersedes_id` is rejected as immutable field change.
- canonical hash for a revision includes explicit `supersedes_id` when present.
- missing historical `supersedes_id` is semantically equivalent to null.

### State / CLI Tests

- `improve propose --supersedes AG-0001` creates revision 1 with that field.
- missing target rejects.
- target not approved rejects.
- target non-materializable rejects.
- target already superseded by another approved current entry emits warning with
  `approval_blocker: true` and still allows proposal.
- approving a second current approved superseder for the same target fails with
  `supersession_fork` until the existing approved superseder is reverted.
- same category/scope without `--supersedes` returns
  `possible_duplicate_active_guidance` warning but still writes proposal.
- same category/scope with `--supersedes` does not emit duplicate warning.
- invalid transition behavior remains unchanged.
- `approve` preserves `supersedes_id` from proposed revision.
- `revert` of superseding entry makes old approved entry materializable again.
- `revert` of superseding entry returns a warning listing re-exposed entries.
- proposed supersession forks are tolerated and warned.
- approved supersession forks are rejected.

### Memory / Runtime Tests

- old approved entry and new approved superseding entry: memory includes only new
  entry.
- chain case `C -> B -> A`: memory includes C; after reverting C, memory includes
  B.
- manifest `materialized_entry_ids` excludes superseded entries.
- rebuild JSON includes skipped superseded diagnostic.
- approved non-materializable superseder suppresses target and emits
  `non_materializable_superseder_suppresses_target`.
- `improve stats --json` reports `superseded_count` and excludes superseded
  entries from `eligible_for_materialization_count`.
- `origin_runtime` remains rendering-only and does not affect materialization.

## Migration Notes

- No ledger rewrite.
- No schema version bump required if `supersedes_id` is optional and historical
  absence remains valid.
- Writer emits explicit `supersedes_id: null` for new entries for deterministic
  shape.
- Add a short migration note explaining that `superseded` is computed, not a
  stored lifecycle status.
- Existing active duplicate entries remain valid; operators may manually create
  a new superseding entry and optionally revert duplicates.
- `supersede != retire`: if an old entry is wrong rather than merely replaced,
  the operator should explicitly revert it.

## Acceptance Criteria

PR4 is complete when:

- `supersedes_id` can be proposed, approved, validated, listed, and shown.
- duplicate same category/scope proposal produces deterministic warning.
- materialization excludes entries suppressed by an approved superseder.
- reverting the superseding entry restores the old approved entry's eligibility.
- reverting a superseder warns about re-exposed entries.
- proposed supersession forks warn with `approval_blocker: true`.
- approved supersession forks are blocked with `supersession_fork`.
- no fifth ledger status is introduced.
- no semantic duplicate detection is introduced.
- `origin_runtime` remains metadata-only.
- `operator_reported_model` remains deferred to v0.7.3 / v0.8 scorecard
  metadata and is not part of v0.7.2.
